## 4. Latency capture recipe (DevTools Performance)

Rajesh asked for stage-by-stage timing so we can target the right bottleneck next cycle. The `lib/perfInstrumentation.ts` utility ships but **is not yet wired into the AI Insights / Ask Pulse pipelines** — that wiring belongs in the same cycle as the actual speedup so the marks measure what's being cut.

For now, Codex can produce a baseline breakdown using **vanilla DevTools** without any code changes:

1. Open `http://127.0.0.1:5174` in the visible browser.
2. Open DevTools → Performance tab.
3. Click record (●).
4. In the app, click into Ask Pulse and submit one of the fixture questions (e.g. `Top 5 sales reps by revenue this year`).
5. Wait until the answer fully renders.
6. Stop recording.
7. In the Performance flame chart, identify the three layers:
   - **Network** (long red bars labelled with `/api/assistant/conversations/...`): backend / Genie / warehouse time.
   - **JS Main thread** during the same window: polling / status UX time.
   - **Layout / Paint** bursts after data lands: frontend render/jank time.
8. Save the screenshot to `docs/evidence/codex-verify-post-uat-<HHMM>/perf-ask-pulse.png`.
9. Repeat for AI Insights (the auto-brief).

Report each layer's measured millisecond cost in your results file. That gives the next cycle a real target instead of guessing.

When the pipeline wiring lands (next cycle), the same recipe will work without DevTools — the marks will show up in the Performance Timings track AND `dumpRun()` will print a `console.table`.
