## Gaps

### P1 — Tooltip Bubble Positioning Clips In Compact Width

Evidence:

- `docs/evidence/tooltip-audit-2026-05-19-1310/001-settings-setup-tooltip-0.png`
- `docs/evidence/tooltip-audit-2026-05-19-1310/002-settings-setup-tooltip-1.png`
- `docs/evidence/tooltip-audit-2026-05-19-1310/004-settings-setup-tooltip-3.png`

The tooltip DOM content exists, but visually the bubble can render partly offscreen or clipped at the left/top of the card in the current visible browser width (`599x694`). Users can lose the beginning of the instruction.

Fix: render HelpTip bubbles through a viewport-aware floating layer or clamp with CSS/JS (`max-inline-size`, collision detection, left/right alignment fallback). This is a visual usability failure, not a data/content failure.

### P1 — Interactive Link Inside Tooltip Is Not Appropriate

Evidence: AI profile tooltip text includes `Databricks docs ↗`.

`HelpTip` is implemented as `role="tooltip"` with `pointer-events: none` on `.pp-helptip__bubble`, and the trigger closes on blur/mouseleave. That means the link inside the tooltip is not reliably clickable. Also, ARIA tooltips should not contain interactive controls.

Fix: either remove the inline link from the tooltip and put a separate visible docs link near the field, or promote this from tooltip to a true popover/dialog pattern with pointer events, focus management, and close behavior.

### P1 — AI Profile Tooltip Does Not Explain The Actual Blocked State

Evidence:

- Tooltip says a profile is a backend the proxy knows how to call.
- Visible setup flow can show `Proxy online · 2 profiles`, but the `AI profile` select remains disabled/unpopulated.
- Related failures in `docs/TOUGH_TEST_RESULTS_2026-05-19-1253.md`: `P3-07`, `P3-08`.

The instruction is conceptually correct but not sufficient when the UI is blocked. If the proxy has profiles but the select is empty, the tooltip should explain whether this is due to allowlist state, `/assistant/profiles` not loaded, or a setup sync gap.

Fix: populate the select from the same source as the proxy health/profiles test, or show a precise inline reason: "Profiles are online but not in the allowlist; open Settings → AI → Provider" / "Profile inventory failed to load; check System → Proxy."

### P2 — Most Settings Subroutes Use Native `title=`, Not Rich Help

Routes checked:

- `/settings/bi`
- `/settings/ai`
- `/settings/preferences`
- `/settings/system`
- `/settings/advanced`

These routes expose many `title=` hints (`Copy link to this section`, layout preset descriptions, readiness chip jump targets), but no rich `HelpTip` triggers. Browser-native titles are weak for mobile, keyboard users, and consistent visual polish.

Fix: keep native titles for tiny utility buttons if needed, but important instructional content should use the same HelpTip/Popover primitive as Setup.

### P2 — Some Native Titles Need Copy Cleanup

Observed examples:

- `Pack (none)` title: `Jump to ai › Pack` should be `Jump to AI → Pack`.
- Root `BI Viz` control still surfaces duplicated `BI BI Viz` text/title context from the switcher issue.
- Layout preset titles in Preferences are useful but long; native titles are a poor container for multi-sentence guidance.

Fix: normalize capitalization/path grammar and move long explanatory copy into visible helper text or rich help.

## Recommendation For Claude

1. Fix HelpTip collision/positioning first. It is the most visible tooltip bug.
2. Remove interactive links from `role="tooltip"` content or replace HelpTip with a true popover variant where links are needed.
3. Add blocked-state guidance for AI profile selection, tied to the same issue causing the empty profile select.
4. Convert important Settings subroute guidance from native `title=` into the shared help primitive.
5. Add regression tests:
   - HelpTip opens on hover/focus/click.
   - Tooltip text is associated with `aria-describedby`.
   - Tooltip bubble clamps within viewport in a compact container.
   - Tooltips do not contain focusable descendants unless rendered as a popover.
   - AI profile blocked state exposes a concrete reason.
