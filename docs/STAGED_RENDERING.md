# Staged Rendering — "1-then-3" Orchestration Spec

> **Status:** Phase D design lock (2026-05-13).
>
> **Sibling spec:** [DISCOVERY_LOOP.md](DISCOVERY_LOOP.md) — the discovery
> output this orchestrator consumes.

## What it is

A **section-by-section** rendering pipeline that fires the HEADLINE generation
first, awaits it, and then fans out the remaining sections in parallel. The
user sees the most important sentence at ~2 seconds; the rest streams in
over the next ~2-3 seconds. Total elapsed time matches today; perceived
performance is dramatically better.

## Why this matters

Today, a structured-sections answer (`HEADLINE / TRENDS / RISKS / ACTIONS /
NEXT`) lands as one LLM response. That means:

- **No first-paint until the full answer is ready** — ~5-6 seconds of blank
  screen
- **No per-section SQL provenance** — the whole answer is "one SQL query",
  so business can't see "which SQL backs the HEADLINE number"
- **No section independence** — if the LLM gets confused mid-answer, the
  whole turn is corrupted
- **No selective re-run** — user can't say "regenerate the RISKS section
  only" without redoing everything

Staged rendering fixes all four.

## The flow

```
T0          User clicks Run with frame=BCG, params={...}
T0+50ms     ONE SQL probe fires (uses cached frame + params)
            ↳ This is the ANALYTICAL query (e.g. "select region, margin, ..."
              with all the filters and aggregates the BCG matrix needs)
            ↳ Generated by Genie OR the analytics-mode SQL-from-LLM path
            ↳ Executed once. Result row(s) cached in conversation memory.
T0+~1s      Probe row(s) returned. Cached.

T0+~1s      Call #1 fires: HEADLINE generation
            ↳ Input: probe row + IR vocabulary + guardrails
            ↳ LLM-only call. No SQL execution. ~500-1000ms.
            ↳ MAY annotate the SQL fragment that produced the HEADLINE number
              ("CTE 'headline_kpi'") for inline provenance.
T0+~2s      HEADLINE renders ← FIRST PAINT

T0+~2s      Calls #2/#3/#4 fire IN PARALLEL: TRENDS / RISKS / ACTIONS
            ↳ Each reads:
               • probe row(s) (same data as HEADLINE)
               • HEADLINE result (so they don't contradict)
               • IR's section-specific instructions
            ↳ Each is LLM-only. ~1-2 sec each.
T0+~3-4s    Each section renders independently as it lands.
            Skeleton placeholders until each lands.

T0+~4s      NEXT computed locally (cheap pattern: "What would you like to
            drill into?" with 2-3 follow-up suggestions derived from
            HEADLINE+RISKS). No LLM call.
```

**Total wall time**: ~4 sec (same as today). **First useful paint**: ~2 sec.

## Architecture — the orchestrator

A new `proxy/lib/sectionedOrchestrator.js`:

```
buildSectionedResponse(profile, ir, request)
   → returns an async iterable that yields:
       { kind: "probe-started", probeId, sql }
       { kind: "probe-completed", probeId, rows, durationMs }
       { kind: "section-started", sectionId }
       { kind: "section-token", sectionId, token }      // when streaming
       { kind: "section-completed", sectionId, body, sql: SectionSql, durationMs }
       { kind: "all-completed", totals }
```

The endpoint becomes **SSE-streaming** so the playground can render sections
as they arrive. SSE is well-supported by Express; the proxy already has
streaming patterns from earlier Pulse work.

```
POST /assistant/conversations/start-sectioned   (Phase D)
Accept: text/event-stream

→ event: probe-started
  data: { probeId, sql }

→ event: probe-completed
  data: { probeId, rows: [...], durationMs }

→ event: section-started
  data: { sectionId: "HEADLINE" }

→ event: section-completed
  data: { sectionId: "HEADLINE", body: "...", sql: { fragment, cteName, … } }

→ event: section-started
  data: { sectionId: "TRENDS" }
   …
```

The legacy non-streaming endpoint stays in place. Staged rendering is opt-in
per-request via the `Accept: text/event-stream` header, and the
playground's AISidebar negotiates it when supported.

## Per-backend behaviour

The orchestrator runs ON TOP of the per-backend translators we shipped in
Phase 11a. The translator emits a payload; the orchestrator decides
**how many calls** to make and **in what order**.

### Genie

Genie is single-message — its API doesn't expose function-calling. So:

- **Call #1 (HEADLINE)**: a Genie conversation `/start` with a user message
  asking for HEADLINE only (the IR's HEADLINE section instruction + the
  cached probe row), wrapped via `genie.translate(ir, ...)` with a narrow
  scope override.
- **Calls #2/#3/#4 (TRENDS/RISKS/ACTIONS)**: `/follow-up` messages on the
  SAME conversation. Genie reuses the loaded space metadata + caches the
  user's question context, so these are cheaper than fresh starts. Each
  asks for ONE section only.
- **Trade-off**: Genie may try to re-execute SQL on the follow-ups. The
  orchestrator passes the cached probe row in the message text ("Based on
  the following result: …") to discourage re-execution. Verify in smoke
  testing.

### Foundation Model / OpenAI / Bedrock-Llama

OpenAI-compatible supports parallel completions + prompt caching natively:

- **Call #1 (HEADLINE)**: chat completion with `response_format` narrowed
  to the HEADLINE section's JSON shape.
- **Calls #2/#3/#4 (parallel)**: three concurrent completions, each with
  `response_format` for its section. **All share the same cached system
  prompt** (Anthropic 5-min cache, OpenAI prompt-cache headers). Input
  token cost is dominated by the system-prompt-once + the small
  per-section delta.

### Bedrock-Anthropic (Claude family)

Same as Foundation Model but uses Anthropic's native message format. Prompt
caching is identical to OpenAI semantically. Translator for this lands in
Phase 11b but the staged orchestrator is ready when it does.

### Supervisor

Supervisor adds an outer fan-out dimension (spaces). The orchestrator runs
**per-space**: each space's Genie path goes through its own 1-then-3
sequence, and the synthesis step (Foundation Model translator with
`task.kind=summarise`) operates on the full per-section bundle from each
space. Synthesis is itself a single call — it doesn't need staging because
its output is the final summary, not a sectioned answer.

## Cost analysis (honest)

| Today (1 call) | Phase D (1 + 3 calls) | Phase D + prompt caching |
|---|---|---|
| 1× system prompt input tokens | 4× system prompt | 1× + 3× cache-hit (≈80% discount) |
| 1× user-message tokens | 4× (each section asks ONE thing) | 4× |
| 1× output tokens (~600 tokens) | ~4×(150 tokens) = same total | same total |
| **Total cost: 1.0×** | **~2.5× without caching** | **~1.3× with caching** |

The ~30% cost premium buys: per-section provenance, selective re-runs,
faster first paint, independence across sections. That trade is acceptable
for analytical workloads; the alternative (the current single-call) is
"cheaper to run but harder to trust."

## SQL Provenance

Per-section SQL provenance is delivered two ways depending on the LLM
backend's capabilities:

### Phase B (cheap — ships first):  CTE-comment markers

The Genie + Foundation Model translators inject `[Section: X]` comment
markers into the SQL-generation prompt asking the LLM to label each CTE
with the section it backs:

```sql
-- Generated by Genie. Section labels added by PulsePlay translator.

WITH
  /* Section: HEADLINE */
  headline_kpi AS (
    SELECT AVG(otif_pct) AS otif_avg, ...
    FROM fct_otif_weekly
    WHERE week >= dateadd(week, -1, current_date)
  ),

  /* Section: TRENDS */
  trends_series AS (
    SELECT week, AVG(otif_pct) AS otif_avg
    FROM fct_otif_weekly
    WHERE week >= dateadd(week, -13, current_date)
    GROUP BY week
  ),

  /* Section: RISKS */
  regional_gap AS (
    SELECT region, AVG(otif_pct) AS otif_avg
    FROM fct_otif_weekly
    WHERE week >= dateadd(week, -1, current_date)
    GROUP BY region
  )

SELECT * FROM headline_kpi
CROSS JOIN trends_series
CROSS JOIN regional_gap;
```

A regex parser splits this into `SectionSql[]` — each section gets its CTE
fragment for the UI. **Works today.** Single Genie call → single SQL →
parsed per section.

### Phase D (proper — ships later): Per-section function calls

When function-calling is available (Foundation Model, OpenAI, Bedrock-Anthropic),
each section is a separate function call against the IR's `functions[]`
catalogue (`compute_kpi`, `decompose_variance`, etc.). Each function call
generates its own SQL execution with its own provenance record. No regex
parsing needed.

Both modes co-exist; the orchestrator picks based on `profile.type`.

## Streaming-aware UI

The playground's AISidebar renders a section list with three states per
section:

| State | Render |
|---|---|
| Pending (`section-started` not received yet) | Hidden |
| Streaming (`section-started` received, no `section-completed`) | Skeleton + spinner |
| Complete (`section-completed` received) | Final rendered content + SQL provenance pill |

Selective re-run UI: each completed section has a "↻ regenerate" button that
fires a new `start-sectioned` with `regenerateOnly: ["SECTION_ID"]`. The
orchestrator replays the cached probe + new LLM call for that section only.

## What CAN'T be done with this approach (limitations)

- **Sections that depend on each other beyond HEADLINE** — e.g. ACTIONS
  that refer to specific RISKS — need the orchestrator to chain
  RISKS→ACTIONS sequentially rather than fanning out. The IR's `output.sections[]`
  schema grows a `dependsOn?: string[]` field in Phase D. Default: every
  section after HEADLINE is independent.
- **Truly free-form answers** (no IR `output.sections[]`) — orchestrator
  degrades to single-call behaviour. Staged rendering is opt-in via the IR
  shape.
- **Sub-second responses** — the section setup overhead (~200ms) makes
  staged rendering inappropriate for trivial questions. The orchestrator
  short-circuits to single-call when the IR has no sections or when
  `request.staged: false`.

## Tests

The orchestrator is the hardest piece in this design. Test coverage must
include:

- **`proxy/tests/sectionedOrchestrator.test.js`** — unit tests with stubbed
  translators verifying call sequencing (HEADLINE before TRENDS/RISKS/ACTIONS),
  parallelism (the 3 follow-ups overlap in wall time), error propagation
  (one section failing doesn't kill the others), and selective re-run.
- **`proxy/tests/sectionedOrchestrator.regression.test.js`** — for each
  authored IR in `pulsepacks/`, run the full pipeline against the legacy
  non-staged path and assert the FINAL ANSWER (concatenation of all sections)
  is semantically equivalent. Catches drift between the two pipelines.
- **`proxy/tests/sseEndpoint.test.js`** — supertest hits the new
  `/start-sectioned` SSE endpoint, parses events, asserts the order
  (probe → headline → parallel sections → all-completed).
- **`playground/src/components/__tests__/SectionedAnswer.test.tsx`** — UI
  rendering of skeleton → streaming → complete transitions, selective
  re-run wiring.

## Open questions

1. **Prompt caching key composition for OpenAI**: the OpenAI Responses API
   caches when the first N tokens match. Our system prompt (translator-generated
   for the IR) is identical across the 4 calls within a single user turn
   — confirmed. But if the user changes parameters mid-session, the system
   prompt is re-emitted with new param values — that defeats caching. Mitigation:
   keep param values out of the system prompt; put them in the user message
   instead. Already true for the current translators (params interpolate
   into the user-message preset sections, not into the IR). Lock this with
   a test in Phase D.
2. **Genie follow-up SQL caching**: Genie's `/follow-up` may regenerate
   SQL even when we pass the cached probe row in the prompt. Smoke before
   committing. If it's a problem, we fall back to single-call for Genie
   and keep staging only for OpenAI/Bedrock/Foundation Model. That's still
   a win for 3 of 4 backends.
3. **NEXT section computation** — described as "local, no LLM". The simple
   heuristic is "if HEADLINE mentioned X, offer to drill into X." Worth
   prototyping with a small rules engine in Phase D.4 rather than another
   LLM call.
